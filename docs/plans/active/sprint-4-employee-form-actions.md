# Sprint 4 Employee Form Actions

## 一、本次目标

Sprint 4 的目标是在现有员工管理前端页面上实现员工新增、编辑、删除交互，让用户可以通过弹窗维护员工基础信息。

本 Sprint 只实现前端员工新增、编辑、删除功能，不实现登录、权限、Excel、批量操作或后端接口开发。

需要完成：

1. 新增员工弹窗。
2. 编辑员工弹窗。
3. 删除确认框。
4. 员工表单校验。
5. 调用后端新增员工接口。
6. 调用后端更新员工接口。
7. 调用后端删除员工接口。
8. 操作成功后刷新员工列表。
9. 操作失败时显示错误提示。
10. 补充必要的前端验证方式。
11. 更新 `docs/progress.md`。

## 二、本次不做什么

Sprint 4 明确不做以下内容：

1. 不实现登录。
2. 不实现权限管理。
3. 不实现 JWT、Session、OAuth 或角色控制。
4. 不实现 Excel 导入。
5. 不实现 Excel 导出。
6. 不实现批量新增员工。
7. 不实现批量编辑员工。
8. 不实现批量删除员工。
9. 不实现员工详情页。
10. 不实现部门新增、编辑、删除。
11. 不修改数据库表结构。
12. 不新增数据库字段。
13. 不修改后端接口，除非发现现有接口 bug。
14. 不新增后端业务接口，除非发现 Sprint Contract 无法通过现有接口完成，并必须先说明原因。
15. 不引入新的前端框架或状态管理库。

## 三、Sprint Contract

Sprint 4 完成标准如下：

1. 必须在现有员工管理页面提供新增员工入口。
2. 必须在员工表格中提供编辑员工入口。
3. 必须在员工表格中提供删除员工入口。
4. 点击新增员工入口后，必须展示新增员工弹窗。
5. 点击编辑员工入口后，必须展示编辑员工弹窗。
6. 编辑弹窗必须使用当前行员工数据回填表单。
7. 新增和编辑可以复用同一个表单弹窗组件或页面内表单逻辑。
8. 新增弹窗标题和编辑弹窗标题必须能区分当前操作。
9. 弹窗关闭或取消时，必须清理表单校验状态和临时表单数据。
10. 员工表单字段必须覆盖 `docs/specs/api.md` 中新增员工请求体字段：`employeeNo`、`name`、`gender`、`phone`、`email`、`departmentId`、`position`、`status`、`hireDate`。
11. `employeeNo` 必须必填。
12. `name` 必须必填。
13. `gender` 必须必选，且只能提交 `0`、`1`、`2`。
14. `status` 必须必选，且只能提交 `1`、`0`。
15. `departmentId` 可以为空；如果选择部门，选项必须来自 `GET /api/departments`。
16. `hireDate` 可以为空；如果填写，提交格式必须与接口文档一致，为 `YYYY-MM-DD`。
17. `phone` 可以为空；如做前端格式校验，应保持宽松，不得阻止后端允许的合法输入。
18. `email` 可以为空；如填写，应进行基础邮箱格式校验。
19. `position` 可以为空。
20. 表单提交前必须执行前端校验。
21. 新增员工必须调用 `POST /api/employees`。
22. 编辑员工必须调用 `PUT /api/employees/{id}`。
23. 删除员工必须调用 `DELETE /api/employees/{id}`。
24. 前端 API 调用必须封装在 `src/api` 下。
25. 页面实现必须放在 `src/views` 下。
26. 通用表单组件如确有必要才放在 `src/components` 下。
27. 前端必须遵守 `src/api -> src/views -> src/components` 分层约束。
28. 页面不得直接使用硬编码后端绝对地址，必须复用现有 `src/api/http.js` 或同层封装。
29. 后端返回 `code != 0` 时，前端必须按失败处理并展示 `message` 或通用错误文案。
30. Axios 网络错误或超时时必须展示通用错误文案。
31. 新增成功后必须关闭弹窗。
32. 新增成功后必须刷新员工列表。
33. 新增成功后应显示成功提示。
34. 编辑成功后必须关闭弹窗。
35. 编辑成功后必须刷新员工列表。
36. 编辑成功后应显示成功提示。
37. 删除前必须展示确认框。
38. 删除确认框必须包含员工姓名或员工编号，避免误删。
39. 用户取消删除时不得调用删除接口。
40. 删除成功后必须刷新员工列表。
41. 删除成功后应显示成功提示。
42. 新增、编辑、删除请求期间应有基础提交态，避免重复提交。
43. 新增、编辑、删除失败时不得清空当前列表数据。
44. 新增或编辑失败时，弹窗应保持打开，便于用户修正。
45. 删除失败时，应显示错误提示，列表保持当前状态。
46. 刷新员工列表时应保留当前筛选条件。
47. 新增成功后的刷新可以回到第 1 页，也可以保持当前页；code-agent 必须选择一种并在 `docs/progress.md` 记录验证结果。
48. 编辑成功后的刷新应保持当前页和当前筛选条件。
49. 删除成功后的刷新应处理当前页删除最后一条记录的情况，不能出现空页但总数仍有上一页数据的异常体验。
50. 状态值 `0` 是有效值，构造表单数据和请求数据时不能因为 falsy 判断误删。
51. 可空字段为空时，提交值必须与后端兼容，不得提交 `undefined`。
52. 不允许在代码、配置或文档中写死真实数据库密码、JWT 密钥或其他敏感信息。
53. 本 Sprint 不允许修改 `docs/design/database.md` 或 `docs/generated/db-schema.md`。
54. 本 Sprint 不允许修改 `docs/specs/api.md` 中已定义的接口语义，除非发现接口文档和后端实现存在明确 bug，并必须在 `docs/progress.md` 说明原因。
55. 前端页面必须能通过 README 中的前端启动方式启动。
56. 必须执行 `cd frontend && npm run build` 并记录结果。
57. 必须在后端可用时进行一次前后端联调验证，并记录结果。
58. 联调验证必须至少覆盖：新增成功、新增失败、编辑成功、编辑失败、删除取消、删除成功、删除失败、操作成功后列表刷新、接口失败错误提示。
59. `docs/progress.md` 必须由 code-agent 更新 Sprint 4 的完成内容、构建结果、联调验证命令、验证结果和遗留问题。

## 四、接口范围

本 Sprint 前端只允许调用以下后端接口：

1. `GET /api/employees`
2. `POST /api/employees`
3. `PUT /api/employees/{id}`
4. `DELETE /api/employees/{id}`
5. `GET /api/departments`

除现有健康检查接口外，不允许新增或修改其他业务接口。

如果 code-agent 发现上述接口存在后端 bug，必须先说明：

1. bug 触发条件。
2. 期望行为。
3. 实际行为。
4. 是否阻塞本 Sprint。
5. 为什么无法只通过前端规避。

## 六、风险点

1. 当前员工列表页面已实现筛选和分页，新增操作列与弹窗时容易破坏 Sprint 3 的列表行为，需要优先保持既有查询、筛选、分页逻辑稳定。
2. 新增、编辑、删除操作都依赖后端统一响应格式，前端不能只依赖 HTTP 状态码。
3. `status = 0` 和 `gender = 0` 都是有效值，表单校验和提交时不能用简单 truthy 判断。
4. `departmentId` 可以为空，前端需要区分未选择部门和选择了具体部门。
5. 日期组件可能返回 `Date` 对象或字符串，提交前必须保证是 `YYYY-MM-DD`。
6. 编辑弹窗回填数据时，列表返回的字段和提交字段接近但不完全等价，不能把 `departmentName` 等只读展示字段误提交给后端。
7. 删除当前页最后一条数据后，如果总数仍大于 0，前端需要避免停留在已经不存在的空页。
8. 新增或编辑失败时如果关闭弹窗，会让用户丢失输入内容；失败时应保留弹窗。
9. 删除失败时如果提前刷新列表，可能掩盖失败原因；应按接口结果决定是否刷新。
10. 重复点击提交按钮可能产生重复请求，尤其是新增员工时可能触发员工编号重复错误。
11. 后端唯一约束会校验 `employeeNo`，前端不应尝试维护本地唯一性判断，只需展示后端错误信息。
12. 部门列表接口失败时，新增和编辑仍可能打开，但部门选择不可用；code-agent 需要决定是否提示并允许提交空部门，或要求重新加载部门。
13. 表单校验规则不能比后端规则过窄，否则可能拦截后端允许的数据。
14. Element Plus 弹窗、表单和日期组件的状态清理不当，可能导致编辑残留到新增表单。
15. 本 Sprint 不做权限控制，因此删除按钮不应引入任何假权限逻辑。
16. 本 Sprint 不修改后端接口；如果出现接口问题，应先定位是否为前端传参格式问题。

## 八、验证方式

自动验证：

1. `cd frontend`
2. `npm run build`

启动验证：

1. 启动后端：

```bash
cd backend
DB_URL=jdbc:mysql://localhost:3306/employee_management DB_USERNAME=root DB_PASSWORD=your_password ./mvnw spring-boot:run
```

2. 启动前端：

```bash
cd frontend
npm run dev
```

3. 打开 Vite 输出的本地地址。
4. 确认员工管理页面可以正常显示。

联调验证：

1. 点击新增员工，弹出新增员工弹窗。
2. 新增弹窗中员工编号或姓名为空时，提交应触发表单校验，不调用新增接口。
3. 填写合法员工信息并提交，应调用 `POST /api/employees`。
4. 新增成功后，弹窗关闭，页面显示成功提示，并刷新员工列表。
5. 使用重复员工编号新增，应展示后端错误提示，弹窗保持打开。
6. 点击某条员工的编辑按钮，弹出编辑员工弹窗并回填该员工数据。
7. 修改员工信息并提交，应调用 `PUT /api/employees/{id}`。
8. 编辑成功后，弹窗关闭，页面显示成功提示，并刷新员工列表。
9. 编辑为重复员工编号或非法数据时，应展示错误提示，弹窗保持打开。
10. 点击删除按钮，应展示删除确认框，确认框包含员工姓名或员工编号。
11. 在删除确认框中点击取消，不应调用 `DELETE /api/employees/{id}`。
12. 在删除确认框中点击确认，应调用 `DELETE /api/employees/{id}`。
13. 删除成功后，页面显示成功提示，并刷新员工列表。
14. 删除不存在或已删除员工时，应展示错误提示，列表保持当前状态。
15. 在筛选条件存在时完成编辑或删除，刷新后应仍按当前筛选条件展示。
16. 后端不可用或接口返回 `code != 0` 时，新增、编辑、删除均应展示错误提示，页面不能崩溃。

说明：

- 验证命令中的 `DB_PASSWORD=your_password` 只能作为占位示例，code-agent 不得把真实密码写入文档或代码。
- 如本机 MySQL 使用临时端口，应按 `docs/progress.md` 最新记录调整 `DB_URL`。
